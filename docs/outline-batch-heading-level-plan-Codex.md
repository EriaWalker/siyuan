# Outline Batch Heading Level Modification Implementation Plan

## 1. Task Goal

Implement an Outline dock/panel feature for selecting multiple heading items and batch-promoting or batch-demoting their heading levels.

Required behavior:

- Selected H2/H3 headings promoted or demoted.
- H1 remains H1 when promoted.
- H6 remains H6 when demoted.
- The operation should preserve current Outline navigation, refresh, document transaction, undo/redo, and editor rendering behavior as much as possible.
- File tree multi-selection should be used as a reference for interaction/state patterns, not copied mechanically.

## 3. File Tree Multi-Selection Logic Extraction

### How multi-selection state is stored

- File tree stores the range anchor in `Files.lastSelectedElement` (`Files.ts` lines 37-42).
- Selected/focused items are represented directly in DOM by `li.b3-list-item--focus`.
- `setCurrent(target)` clears all `li.b3-list-item--focus` under the file tree and applies the class to one item (`Files.ts` lines 1212-1225).
- Shift selection uses the anchor plus current DOM order; it does not maintain a separate `Set` of IDs (`Files.ts` lines 347-376).

### How click, right-click, and keyboard selection work

- Normal click on an `LI` clears previous focus via `setCurrent()` and opens/toggles the document/notebook (`Files.ts` lines 377-430).
- Ctrl/meta click toggles `b3-list-item--focus` on the clicked item and updates `lastSelectedElement` (`Files.ts` lines 343-346).
- Shift click validates or rebuilds the anchor, clears existing focus, collects all `li.b3-list-item`, computes start/end indexes, and applies `b3-list-item--focus` to the range (`Files.ts` lines 347-376).
- Right-click is handled globally in `app/src/menus/index.ts` lines 25-83. For `data-type="navigation-root"` and `data-type="navigation-file"`, it opens `initNavigationMenu()` or `initFileMenu()`.
- `initNavigationMenu()` and `initFileMenu()` preserve an existing multi-selection if the right-clicked item is already focused; otherwise they clear focus and focus the right-clicked item (`app/src/menus/navigation.ts` lines 194-212 and 428-445).
- Keyboard context menu uses the current focused file-tree item from `boot/globalEvent/keydown.ts` lines 681-686.

### How DOM classes mark selected state

- File tree uses `b3-list-item--focus` as both the current item and multi-selected item marker.
- It also clears `select-start` and `select-end` markers after item clicks (`Files.ts` lines 432-433), and the menu setup clears those markers when replacing selection (`navigation.ts` lines 201-207 and 435-441).
- No app-wide use of `b3-list-item--selected` was found by repository search. If Outline introduces that class, CSS support must be added or confirmed.

### How menu actions obtain multiple targets

- `initMultiMenu(selectItemElements, app)` receives a `NodeListOf<Element>` selected from `.b3-list-item--focus` (`navigation.ts` lines 34-52).
- It extracts block IDs from `data-node-id` (`navigation.ts` lines 44-50).
- It passes selected elements to operations such as copy, duplicate, move, delete, add-to-database, export, and plugin event details (`navigation.ts` lines 53-190).
- Single-file menu still uses `fileElement.querySelectorAll(".b3-list-item--focus")` for operations that may act on preserved selection, such as move/delete (`navigation.ts` lines 520-538).

### Which logic can be migrated to Outline

- Migrate the concepts: range anchor, DOM-order range selection, right-click selection preservation, and collecting selected targets from current rendered Outline items.
- Do not migrate file path/notebook logic, file drag-and-drop logic, or file menu commands.
- Prefer an Outline-specific state layer so current-heading focus remains independent from batch selection.

## 5. Correct Technical Path for Batch Heading Level Modification

### Recommended path

Use a single Protyle transaction containing per-heading `update` operations generated from current block DOM and per-heading target levels.

Recommended flow:

1. Collect selected Outline heading IDs and their current levels from `li.b3-list-item[data-type="NodeHeading"][data-subtype]`.
2. Compute target levels independently:
   - Promote: `Math.max(1, currentLevel - 1)`.
   - Demote: `Math.min(6, currentLevel + 1)`.
3. Filter unchanged headings, e.g. H1 on promote and H6 on demote.
4. Fetch current DOM for changed IDs with `/api/block/getBlockDOMs`, defined in `kernel/api/router.go` lines 197-198 and implemented in `kernel/api/block.go` lines 766-798.
5. For each returned heading DOM, use `protyle.lute.Blocks2Hs(dom, targetLevel)` to generate the updated DOM. This mirrors the existing transformation mechanism used by `turnsIntoTransaction()` at `transaction.ts` lines 1156-1217, but allows different target levels per heading.
6. Build one `doOperations` array and one `undoOperations` array with `action: "update"`, `id`, and `data`.
7. Update any currently loaded editor DOM nodes by `data-node-id` and rerender math for replaced nodes, following the existing pattern in `Outline.genHeadingTransform()` lines 1329-1348.
8. Call `transaction(protyle, doOperations, undoOperations)` once so undo/redo is one logical batch operation and Outline refresh continues through `onTransaction()`.

### Why other paths should not be used

- Do not call `/api/block/getHeadingLevelTransaction` as the default batch path. `kernel/model/block.go` lines 759-817 show that it changes the selected heading and heading-child headings by a shared diff, clamped to H1-H6. That is correct for “with subtitle” behavior but not for exact selected-heading-only batch promotion/demotion.
- Do not call `turnsIntoTransaction()` directly for a mixed H2/H3 selection because it accepts a single `level` option for all selected blocks (`transaction.ts` lines 1097-1105 and 1156-1159). It can convert many blocks to one common level, but it cannot express H2->H1 and H3->H2 in one call.
- Do not use `/api/block/updateBlock` or `/api/block/batchUpdateBlock` for this feature. Those endpoints update block content from markdown/DOM API inputs and bypass the existing editor transaction helper behavior that Outline and Protyle already rely on for undo and refresh. Their routes are in `kernel/api/router.go` lines 230-231.
- Do not use `/api/attr/setBlockAttrs`; heading level is structural block DOM/AST state (`data-subtype`, `HeadingLevel`), not an IAL attribute. The route is in `kernel/api/router.go` line 271.

### How to handle multiple heading blocks

- Batch all changed headings into one transaction when possible.
- Preserve selected order by Outline DOM order, not by click order, for stable behavior.
- If selected headings include parent/child heading rows, only update explicitly selected heading blocks. Do not automatically include unselected descendants.
- If a selected heading DOM cannot be fetched or is no longer `NodeHeading`, skip it and optionally show a message after the transaction.

### Boundary handling

- Promote H1: target level remains 1; exclude it from `doOperations`.
- Demote H6: target level remains 6; exclude it from `doOperations`.
- Mixed selection:
  - H1, H2, H6 promoted -> H1 unchanged, H2 -> H1, H6 -> H5.
  - H1, H2, H6 demoted -> H1 -> H2, H2 -> H3, H6 unchanged.
- If all selected headings are unchanged for the requested direction, the menu item should be disabled or no-op with no transaction.

## 6. Suggested Data Structure and State Design

### Where Outline multi-selection state should be stored

- Add Outline-local state, not global state:
  - `private selectedHeadingIds = new Set<string>();`
  - `private lastSelectedElement: HTMLElement | null = null;`
- The `Set` protects selection from `Tree.setCurrent()` clearing DOM classes during click dispatch.
- DOM classes should be reapplied from the `Set` after `Tree.updateData()` because Outline refresh rebuilds the tree (`Outline.update()` lines 665-694).

### Whether to reuse the file tree selected/focus design

- Reuse the selection concepts but not the exact class contract.
- File tree can use `b3-list-item--focus` for selection because file-tree focus is not also a live document-heading sync marker.
- Outline should preserve `b3-list-item--focus` for current-heading sync and use a separate selection marker such as `b3-list-item--selected` or `data-selected="true"`.
- Repository search found no existing `b3-list-item--selected` usage. Unconfirmed item: confirm the desired CSS class and styling location before implementation.

### Whether an independent helper is needed

Recommended helper methods inside `Outline`:

- `getSelectableOutlineItems(): HTMLElement[]` returns visible/rendered heading `li.b3-list-item[data-type="NodeHeading"][data-node-id]`.
- `getSelectedOutlineItems(fallback?: HTMLElement): HTMLElement[]` returns selected rows by `selectedHeadingIds`; if empty, returns the fallback row.
- `applyOutlineSelectionClasses()` syncs `selectedHeadingIds` to DOM.
- `replaceOutlineSelection(element: HTMLElement)` clears the set and selects one row.
- `toggleOutlineSelection(element: HTMLElement)` toggles one row and updates `lastSelectedElement`.
- `selectOutlineRange(toElement: HTMLElement)` uses DOM order and `lastSelectedElement`.
- `batchChangeHeadingLevel(direction: "promote" | "demote", sourceElement: HTMLElement)` builds and commits the transaction.

### Whether new type definitions are needed

- No global type definition is strictly required if helpers remain private in `Outline.ts`.
- A small local type is useful for readability:
  - `type TOutlineHeadingLevelDirection = "promote" | "demote";`
  - `interface IOutlineHeadingLevelUpdate { id: string; currentLevel: number; targetLevel: number; element?: HTMLElement }`
- If helper logic is extracted to `transaction.ts`, define a focused exported function signature instead of broad new global types.

## 8. Edge Cases

- Promoting H1:
  - Keep H1 unchanged; do not generate an update operation.
  - If all selected headings are H1, disable promote or no-op.
- Demoting H6:
  - Keep H6 unchanged; do not generate an update operation.
  - If all selected headings are H6, disable demote or no-op.
- Selecting H1, H2, and H6 at the same time:
  - Promote: H1 unchanged, H2 -> H1, H6 -> H5.
  - Demote: H1 -> H2, H2 -> H3, H6 unchanged.
- Selecting non-heading items:
  - Outline should only select `data-type="NodeHeading"` rows for this feature.
  - If a non-heading row is encountered, skip it and do not include it in menu applicability checks.
- Empty selection:
  - Treat the right-clicked heading as the fallback selection.
  - If no fallback heading exists, do not show enabled batch commands.
- Selected item points to a non-existing block:
  - `/api/block/getBlockDOMs` may return no DOM or an empty DOM. Skip that ID and optionally show a concise message.
  - If every selected target is missing, do not call `transaction()`.
- Selection after document refresh:
  - `Outline.update()` rebuilds the tree; reapply selected classes from `selectedHeadingIds` after `this.tree.updateData(data.data)`.
  - Remove IDs from `selectedHeadingIds` if their row no longer exists after refresh.
  - Preserve existing current-heading focus behavior from `currentId` at `Outline.ts` lines 665-692.
- Outline refresh after modification:
  - Rely on the existing transaction -> `savedoc` -> `onTransaction()` -> `/api/outline/getDocOutline` flow.
  - Avoid manually re-fetching Outline immediately unless WebSocket refresh proves insufficient.
- Undo/redo:
  - Required. Use one `transaction(protyle, doOperations, undoOperations)` call so `protyle.undo.add()` records one batch undo item (`transaction.ts` lines 1390-1399).
- Folded headings:
  - If a changed heading is folded, existing `turnsIntoTransaction()` has special fold handling when heading level changes (`transaction.ts` lines 1195-1199). The batch helper should either replicate this specific fold handling or call a shared helper that does. Unconfirmed item: verify desired behavior for folded selected headings before implementation.
- Selected parent and selected child heading:
  - Since the operation changes exact selected heading blocks, update both if both are explicitly selected.
  - Do not use `getHeadingLevelTransaction` because it may also alter unselected descendants.
- Readonly or publish mode:
  - Do not show editable actions when `window.siyuan.config.readonly` is true, matching existing `showContextMenu()` behavior.
